# Spanish Pronunciation Assessment

A research-grade pipeline for **automatic phonological competence assessment in Spanish as a Foreign Language (ELE)**, positioned at the intersection of speech processing, L2 phonology, and computer-assisted language learning (CALL/CAPT).

> **Status:** active research prototype. The architecture described below reflects the target design; individual modules are at varying stages of implementation. See [Implementation status](#implementation-status) for details.

---

## Conceptual design

### The core problem this tool addresses

Conventional pronunciation assessment systems rely on Automatic Speech Recognition (ASR) engines to transcribe what a learner said, then compare the transcript to the expected text. This approach has a structural flaw: ASR models are trained to *understand despite errors*. When a learner mispronounces a phoneme, the ASR engine infers the correct word from context and discards precisely the phonological deviation that needs to be detected.

This tool bypasses that flaw by using **Goodness of Pronunciation (GOP)** — a per-phoneme probabilistic score computed directly from acoustic posteriors, independently of word recognition. GOP asks not *"what did you say?"* but *"how faithfully did you produce this specific phoneme?"* (Witt & Young, 2000).

### The same-speaker architecture

The default use case is **same-speaker mode**: both audio files — the reference and the evaluated recording — are produced by the same voice, under the same recording conditions.

The reference is not a generic native speaker. It is generated by **HeyGen**, a high-quality voice-cloning tool, from the learner's own voice. The result is a phonologically correct Spanish recording that shares the exact timbre, resonance, and formant space of the learner's voice.

This design has three direct consequences for the measurement pipeline:

1. **Inter-speaker variability is zero by construction.** All acoustic deltas (F0, intensity, duration, formants) reflect differences in *production*, not differences in anatomy. No inter-speaker normalization is needed or appropriate.
2. **GOP scores are calibrated against the learner's own voice.** A low GOP score means the learner's production diverges from the best possible rendering of *their own voice speaking correct Spanish* — not from an unknown native speaker.
3. **Vowel formant analysis becomes tractable.** Since the baseline formant space (F1/F2) is identical in both recordings, deviations directly signal vocalic substitutions (e.g. /e/ → /i/, /o/ → /u/).

### Module 3 in detail: speaker-adapted GOP

The GOP module is the architectural centerpiece. It departs from the heuristic delta approach used in earlier versions of this pipeline.

**Base model:** [`carlosdanielhernandezmena/wav2vec2-large-xlsr-53-spanish-ep5-944h`](https://huggingface.co/carlosdanielhernandezmena/wav2vec2-large-xlsr-53-spanish-ep5-944h) — a wav2vec2-large model fine-tuned on ~944 hours of native Spanish speech. This model encodes the phonological space of native Spanish production.

**Speaker adaptation:** The base model is further fine-tuned via **LoRA** (Low-Rank Adaptation, Hu et al., 2022) on the learner's HeyGen recordings. LoRA modifies only ~1% of the model's parameters, which is proportionate to the available adaptation data (a few minutes of speech per text) and prevents catastrophic forgetting of the native phonological representations acquired on 944 hours.

The resulting LoRA adapter (~a few MB) is stored separately from the base model and can be swapped per learner without duplicating the 1.2 GB base weights.

**GOP variant:** The implemented module currently computes standard phone-level GOP from the adapted wav2vec2 CTC frame logits. CaGOP-style transition and attention-weighted duration factors remain research roadmap items rather than active production logic.

**GOP formula (standard, Witt & Young 2000):**
```
GOP(p) = log P(p | o_t1..t2) − max_q log P(q | o_t1..t2)
         normalized by phoneme duration
```

Scores are computed from **raw logits** (not softmax-normalized probabilities), following Shi et al. (2022), which improves discriminability for mispronunciation detection.

**Device support:**
```python
# Automatic backend detection
if torch.cuda.is_available():       device = "cuda"
elif torch.backends.mps.is_available(): device = "mps"   # Apple Silicon
else:                               device = "cpu"

# TrainingArguments precision flags
fp16 = torch.cuda.is_available()          # CUDA only
bf16 = torch.backends.mps.is_available() # Apple Silicon — bfloat16 native
```
The fine-tuning step falls back to CPU automatically if an MPS-incompatible CTC operation is encountered, with an explicit warning.

## What is measured vs what is heuristic

| Layer | What is measured | What is heuristic / calibration-pending |
|---|---|---|
| ASR gate | Whisper transcripts vs shared script | Warn/reject thresholds |
| MFA alignment | Phoneme timestamps for both recordings | MFA aligns to expected transcript; does not verify what was actually produced |
| GOP (segmental) | Per-phoneme posterior probability ratio vs speaker-adapted model | Error/warning thresholds (GOP_ERROR_THRESHOLD, GOP_WARNING_THRESHOLD) — empirically calibrated via controlled error injection |
| Prosody | F0, intensity, pauses inside MFA-aligned intervals | Raw intensity is sensitive to gain and microphone distance — mitigated by same-speaker RMS normalization |
| Lexical stress | silabeador expected pattern vs acoustic prominence | Syllable windows proportional to grapheme length, not true signal syllable boundaries |
| Scoring weights | Six-domain weighted aggregate | Domain weights and penalties are research hypotheses pending expert validation |

## Technical validity vs pedagogical validity

This distinction is central to the project's research agenda.

**Technical validity** means the GOP score correctly detects and quantifies acoustic deviations from the target phoneme. This is achievable with a well-calibrated acoustic model.

**Pedagogical validity** means the score predicts what a qualified ELE teacher would judge as communicatively significant. These two measures can diverge in important ways:

- **Perceptual redundancy:** a phoneme deviation detectable by the machine may be entirely compensated by lexical context in human perception (e.g. the French /ʁ/ replacing Spanish /r/ is universally detected by GOP but rarely impedes intelligibility in context).
- **Functional asymmetry:** not all phonemes carry equal communicative weight. The /b/–/v/ non-distinction in Spanish is phonologically non-pertinent; a low GOP score on this point would be pedagogically misleading for French learners.
- **Supra-segmental blindness:** GOP operates phoneme by phoneme and cannot detect syllable structure errors (e.g. French learners breaking Spanish synérèse), which native teachers identify as the most salient foreign accent marker.

Bridging this gap requires a **human expert validation phase**: ELE teachers rate the same learner recordings the system evaluates, and their judgments are used to calibrate domain weights and GOP thresholds per phoneme class and per L1 profile.

## Score model

Six domains, currently weighted by research hypothesis. Weights and penalties will be revised after expert validation.

| Domain | Max points | Scoring basis |
|---|---|---|
| `segmental_precision` | 30 | GOP scores per phoneme (replaces duration deltas) |
| `syllabic_structure_and_articulation` | 15 | MFA duration distributions + synérèse detection |
| `lexical_stress` | 20 | silabeador expectation vs acoustic prominence |
| `sentence_prosody` | 20 | F0 contour delta vs HeyGen reference |
| `fluency_and_voice_control` | 10 | Pause rates, tempo, voice onset |
| `phonological_intelligibility` | 5 | Aggregate penalty on high-impact deviations |

All thresholds and penalties are externalized in `src/config.py` and configurable via `.env`. No magic numbers in scoring logic.

## CLI

```bash
python -m src.pipeline \
  --reference-audio data/heygen_reference/text_001.wav \
  --audio data/learner/text_001.wav \
  --text "El mismo guion leído por los dos." \
  --output data/output/report.json
```

Flags:
- `--strict-text-match` — raises ASR warn/reject thresholds
- `--allow-partial-match` — loosens ASR gate only
- `--force-gop-retrain` — re-runs LoRA speaker adaptation even if an adapter exists
- `--skip-gop` — disables GOP for this run and keeps heuristic fallback scoring
- `--gop-debug-output path/to/gop.json` — exports raw GOP payload for calibration
- `--debug` — prints full tracebacks

## GOP validation workflow

The next research priority is to prove and calibrate the speaker-adapted GOP signal before expanding the learner-facing product surface.

1. Prepare a controlled local corpus using `data/validation_manifest.example.json` as a template.
2. Run the pipeline with `--gop-debug-output` for HeyGen controls and learner/error samples.
3. Aggregate GOP distributions:

```bash
python scripts/analyze_gop_calibration.py \
  data/output/*.gop.json \
  --manifest data/validation_manifest.json \
  --output data/output/gop_calibration_summary.json
```

4. Export machine reports for ELE teacher annotation:

```bash
python scripts/export_expert_validation.py \
  data/output/*.report.json \
  --output data/output/expert_validation.csv
```

The calibration script estimates starter `warning` and `error` thresholds only when the manifest labels samples as `expected_outcome=correct` and `expected_outcome=error`. Real audio files should remain local unless the speaker consent and sharing policy are explicit.
